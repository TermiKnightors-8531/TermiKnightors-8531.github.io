Useful things to know about git

Git Terms

Repository:
a location for storing code on github> You can contribute and get code from here
Branch: a location for all of your code. The master branch is the default branch, however creating new branches is a good idea when testing out a drastic change. Branches are a safe way to share sketchy code without murdering our function master branch!

Where it all Starts

The Command Pallete

To open the command pallete in VSCode, press the buttons CTL+SHIT+P. This should open something that looks similar to the image below.

Here, we can access all of the commands necesarry to run git and also deploy our robot code when it becomes necesarry. To access git commands, open the pallete and type in the git. This will open up many different suggestions. We'll address what they all mean later.

Commits

What is a commit?

When you commit to git, you are publishing your changes to everyone in the repository. Normally you will commit at the end of a coding session or at a land mark. Try not to dump massive amounts of code in one commit. Commits should really just nice chunks of what changes you made.

What is a commit message

A commit message is a description of the changes that you made. It is a short two to six-ish word summary of what you did. Examples of good commit messages are: "added button mapping" , "fixed drive error", "added arm functions", etc. Don't be like me two years ago. Some bad, but real commit messages are things like "dump", "dead computer", "aaaaaah", "idk man", etc. PLEASE, don't be stupid like me...

When should I commit?

At the end of a meeting or after a major change, it's always good to commit. Whenever you log off your computer for the day after coding, commit your changes.

How do I commit?

Although you can use the command pallete, it's much easier to the use the spaghetti and meatball icon.

Once you've opened this icon, if you've modified files, you'll see them at the right. To add a file to your next commit, click the plus arrow next to it. When you commit, try to group files based on the changes that you have had. If you've changed code for the arm in two files, add those files to your commit. Then, add a commit message to describe what you did.

You can hit the purple commit checkmark. Repeat this process for all of the groups of files that you are going to commit. Then, a button will appear that allows you to sync changes. Click this and then you've completed all of your commits!

Branches

What is a branch?

A branch is a place to store code within our repository. It is a way for us to work on code in parallel without murdering our master branch. The master branch is a location where our main code lives. For example of branches, if one of us is working on the arm code, and another is working on button mapping, this is a great time to use branches. The development can happen independently without conflicts. When ready, we can merge these changes into our master branch.

How do I create a branch?

To navigate and move between branches, open the command pallete. To create a new branch, type git: create branch from... into the command pallete. Enter a name for the new branch. Normally, this name should be a short and descriptive one. Things like "arm test" or "button mapping" are good options. After this, you will most often create the branch from the main one.

Once you have finished and are ready to commit, you must first publish the branch. There will be a button in the spaghetti and meatball icon to do so!

How do I change branches?

Enter the command pallete and type git: checkout to. From here, you can select the branch in which you want to make changes.
